# SQL Transactions Demo Service

Демо-сервис интернет-магазина на Python + PostgreSQL, который показывает 3 транзакционных сценария:

1. Размещение заказа
2. Обновление email клиента
3. Добавление нового продукта

## Технологии

- Python 3.12
- SQLAlchemy 2.x
- psycopg2
- PostgreSQL 16
- Docker / Docker Compose

## Что делает сервис

### Сценарий 1 — Размещение заказа

В одной транзакции:
- создается `orders` с `totalamount = 0`
- добавляются строки в `orderitems` (`subtotal = price * quantity`)
- пересчитывается `totalamount` как сумма всех `subtotal`
- при любой ошибке выполняется `rollback`

### Сценарий 2 — Обновление email клиента

В одной транзакции:
- проверяется корректность email
- проверяется уникальность нового email
- обновляется email по `customerid`
- при ошибке выполняется `rollback`

### Сценарий 3 — Добавление продукта

В одной транзакции:
- проверяется, что `productname` не пустой
- проверяется, что `price > 0`
- добавляется продукт
- при ошибке выполняется `rollback`

## Быстрый старт

1. Перейдите в папку проекта:

```bash
cd "project"
```

2. Запустите сервисы:

```bash
docker compose up --build
```

3. В консоли появятся:
- логи подключения к БД и транзакций
- результаты `SELECT` после каждого сценария

## Остановка

```bash

```

```bash

```

## Логирование

Логирование настроено через `logging` в `app/main.py`.
Формат:

```text

```

Уровень можно задать переменной окружения:

```env
LOG_LEVEL=INFO
```

## Как объяснить проект на защите

### 1) Что сказать в начале (30 секунд)

"Это демонстрационный сервис интернет-магазина с тремя SQL-транзакциями.  
Главная идея: либо все шаги операции сохраняются (`commit`), либо при любой ошибке все изменения откатываются (`rollback`)."

### 2) Где в коде начинается транзакция

Открыть `app/transactions.py` и показать:
- `with session.begin():` — явное начало транзакции
- выход из блока без ошибки -> `commit`
- исключение внутри блока -> `rollback`

Короткая фраза для преподавателя:

"Транзакции начинаются явно в `with session.begin():`.  
Это гарантирует атомарность операции: все шаги как единое целое."

### 3) Как показывать демо в терминале

Запуск:

```bash
docker compose down -v
docker compose up --build
```

Почему `down -v`: чтобы база была чистой и не оставались данные с прошлого запуска.

Дальше по логам:
- `shop-postgres ... ready to accept connections` -> БД готова
- `app.database ... База данных готова к подключению` -> сервис дождался БД
- `Сценарий 1...` -> размещение заказа
- `Сценарий 2...` -> обновление email
- `Сценарий 3...` -> добавление продукта
- `Все сценарии успешно выполнены` -> демо прошло полностью

### 4) Как объяснить частые вопросы

`Почему раньше была ошибка duplicate key?`  
"Потому что в БД уже был товар с таким названием из прошлого запуска. У нас уникальное ограничение на `productname`, поэтому вставка отклоняется и транзакция откатывается."

`Почему total_amount может быть 0.00, если есть позиции?`  
"Это бывает, если сумму посчитать до синхронизации pending-изменений. В коде добавлен `session.flush()` перед суммированием, поэтому теперь итог считается корректно."

### 5) Короткий итог для финала защиты

"В работе реализованы три атомарные транзакции, обработка ошибок с rollback, логирование, connection pool и запуск через Docker Compose с автоматической инициализацией БД."
