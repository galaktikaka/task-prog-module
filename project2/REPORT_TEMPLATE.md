# Отчет по практике: RabbitMQ vs Redis

## 1. Цель и условия эксперимента

Сравнить `RabbitMQ` и `Redis` как message broker в одинаковых условиях:

- одинаковый формат сообщения (JSON);
- одинаковые `producers/consumers`;
- одинаковая длительность прогона;
- одинаковые лимиты ресурсов (single instance).

Параметры:

- размеры сообщений: `128 B`, `1 KB`, `10 KB`, `100 KB`;
- интенсивность: `1000`, `5000`, `10000 msg/s`;
- длительность: `30 s` (или ваше значение).

## 2. Стенд

- Producer -> Broker -> Consumer
- Broker по очереди:
  - RabbitMQ (single instance);
  - Redis (single instance).

Инструмент: собственный Python load-generator (`src/benchmark.py`).

## 3. Метрики

- `messages/sec` (sent/received);
- `avg latency`, `p95`, `max`;
- `sent`, `received`, `lost`, `send_errors`;
- `backlog` в конце;
- точка деградации.

## 5. Анализ

### 5.1 Базовое сравнение

- Кто отправил/обработал больше сообщений при одинаковых условиях.
- Где меньше потерь и ошибок.

### 5.2 Влияние размера сообщения

- На маленьких payload (`128B`, `1KB`) — кто быстрее.
- На больших payload (`10KB`, `100KB`) — у кого сильнее просадка throughput и рост latency.

### 5.3 Влияние интенсивности

- При каком `msg/s` впервые появляется `degraded=true`.
- Что стало причиной: backlog, latency, ошибки или потери.
