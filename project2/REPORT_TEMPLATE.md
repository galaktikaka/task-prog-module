# Отчет по практике: RabbitMQ vs Redis

## 1. Цель и условия эксперимента

Сравнить `RabbitMQ` и `Redis` как message broker в одинаковых условиях:

- одинаковый формат сообщения (JSON);
- одинаковые `producers/consumers`;
- одинаковая длительность прогона;
- одинаковые лимиты ресурсов (single instance).

Фактически выполненные параметры (quick-прогон):

- размеры сообщений: `128 B`, `1 KB`;
- интенсивность: `1000`, `5000 msg/s`;
- длительность: `10 s`;
- `producers=2`, `consumers=2`.

## 2. Стенд

- Producer -> Broker -> Consumer
- Broker по очереди:
  - RabbitMQ (single instance);
  - Redis (single instance).

Инструмент: собственный Python load-generator (`src/benchmark.py`).

## 3. Метрики

- `messages/sec` (sent/received);
- `avg latency`, `p95`, `max`;
- `sent`, `received`, `lost`, `send_errors`;
- `backlog` в конце;
- точка деградации.

## 5. Анализ

### 5.1 Базовое сравнение

- Оба брокера отработали стабильно: `lost=0`, `send_errors=0`, `backlog=0` во всех кейсах.
- В quick-нагрузке Redis обработал немного больше сообщений при `5000 msg/s`:
  - RabbitMQ: ~`2820-2830 msg/s`;
  - Redis: ~`2987-3000 msg/s`.

### 5.2 Влияние размера сообщения

- В quick-прогоне сравнивались только `128B` и `1KB`.
- Для обоих брокеров заметной деградации при переходе с `128B` на `1KB` не зафиксировано.
- По latency Redis в этих условиях показывает более низкие `avg` и `p95`.
- Для анализа больших payload (`10KB`, `100KB`) нужен полный прогон.

### 5.3 Влияние интенсивности

- При повышении интенсивности с `1000` до `5000 msg/s` деградация не наступила (`degraded=False` во всех 8 кейсах).
- Причины деградации не выявлены: нет роста backlog, нет ошибок, нет потерь, `p95` значительно ниже порога `200 ms`.

## 6. Итоговые выводы

1. В quick-сценарии большую пропускную способность показал `Redis` (особенно на `5000 msg/s`).
2. На payload `128B-1KB` оба брокера работают стабильно; заметной деградации от роста размера не обнаружено.
3. Точка деградации single instance в quick-наборе не достигнута:
   - RabbitMQ: деградация не зафиксирована до `5000 msg/s` при `1KB`;
   - Redis: деградация не зафиксирована до `5000 msg/s` при `1KB`.
4. Чтобы строго выполнить пункт про деградацию на высокой нагрузке, требуется полный прогон (`10KB`, `100KB`, `10000 msg/s`).
5. Выбранный инструмент (`src/benchmark.py`) подходит, потому что:
   - одинаковая логика для обоих брокеров;
   - повторяемые прогоны;
   - автоматический сбор метрик в CSV/MD.
